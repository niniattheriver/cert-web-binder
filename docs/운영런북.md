# 운영 런북

내부망에 설치된 인증 근거자료 웹 바인더의 일상 운영 절차입니다.

> **비개발자 담당자용 요약**: 백업·점검·체크리스트는 화면에서 전부 할 수 있습니다 —
> **관리자로 로그인 → 오른쪽 위 사람 아이콘 → [운영 점검 (백업·점검)]**.
> 아래 문서의 `GET/POST /api/...` 표기는 IT 담당자·자동화용 참고이며 몰라도 됩니다.

## 서비스 재시작

- **Linux(systemd)**
  - 재시작: `sudo systemctl restart qaevidence`
  - 상태: `systemctl status qaevidence`
  - 로그: `journalctl -u qaevidence -e`
- **Windows(WinSW)**
  - 재시작: `qaevidence-service.exe restart` (중지 `stop` · 시작 `start` · 상태 `status`)
  - 로그: 설치 루트의 `qaevidence-service.out.log` / `qaevidence-service.err.log`
- **수동(점검용)**: 설치 루트에서 `NODE_ENV=production npm start`

## data/ 위치와 백업

모든 상태는 설치 루트의 **`data/`** 에 있습니다:

- `app.db` (+ `-wal` / `-shm`) — 문항·근거·채점·이력 등 전체 도메인 데이터(SQLite WAL).
- `files/` — 근거 PDF·이미지. **내용주소(sha256) 불변** 저장.
- `backups/` — 자동/즉시 백업 스냅샷.
- `session-secret.txt` · `initial-admin-password.txt`(최초 1회, 사용 후 삭제).

백업 방법:

> **⚠️ 내장 백업의 범위**: 자동 백업과 즉시 백업 ZIP은 **DB 중심**입니다 — 첨부·지침서 원본이
> 있는 `files/` 폴더 본문은 포함되지 않습니다(ZIP에는 파일 목록·해시만 들어갑니다).
> **완전 복구에는 DB 백업 + `files/` 폴더가 모두 필요**하므로, 아래 "오프라인 복사"로
> `data/` 전체를 다른 장비·디스크에 주기적으로 복사해 두는 것을 강력히 권장합니다.

- **자동 백업** — 앱이 매일 **03:00** 에 `data/backups/app-YYYYMMDD.db`(VACUUM INTO) 생성. **일 30개 + 월 12개** 보존. 별도 cron 설정 불필요(서비스가 상시 구동되어 있으면 됩니다). `pre-import-*.db`(가져오기 직전 스냅샷)와 수동 `backup-*.zip`은 자동 정리되지 않으므로 오래된 것은 확인 후 직접 지워 주세요.
- **즉시 백업 ZIP** — 화면에서: **운영 점검 → [지금 백업]** 버튼 → `data/backups/backup-….zip` 생성(DB 스냅샷 + 파일 매니페스트 — 파일 본문 미포함). (IT 참고: `POST /api/admin/backup`)
- **오프라인 복사(회사 절차, 한 줄)** — `data` 폴더를 통째로 다른 장비·USB로 복사합니다.
  탐색기에서 설치 폴더의 **`data` 폴더를 복사 → 백업 장치에 붙여넣기** 하면 됩니다(가장 쉬운 방법).
  - IT 참고(자동화): Windows `robocopy C:\qaevidence\data D:\backup\qaevidence-data /MIR` ·
    Linux `rsync -a --delete /opt/qaevidence/data/ /backup/qaevidence-data/`
- **복원** — 서비스 중지 → `data/` 폴더를 되돌림 → 서비스 시작. (세션 시크릿·파일이 함께 따라옵니다.)

> WAL 파일(`-wal`/`-shm`)까지 함께 복사해야 최신 상태가 보존됩니다. 가능하면 서비스 중지 후 복사하거나 즉시 백업 ZIP을 사용하세요.

## 사용자 추가

- `admin` 계정으로 로그인 → 관리 화면에서 사용자 생성.
- 역할: `admin`(관리) · `editor`(편집) · `viewer`(열람 전용 — 심사위원).
- viewer 임시계정은 만료일(`expires_at`)을 둘 수 있습니다(심사 종료 후 자동 비활성).
- 권한 규약: 인증 GET은 로그인 필수, 데이터 변경은 `editor` 이상.

## 관리 상태 신호

**운영 점검** 화면(관리자)이 아래 신호를 보여 줍니다. 빨간(경고) 신호의 의미와 조치:
(IT 참고: `GET /api/admin/status`)

| 신호 | 의미 | 조치 |
|------|------|------|
| **저장 공간 경고(빨강)** | `data/` 볼륨 여유공간 부족 | 오래된 `backups/` 를 다른 장비로 이관 후 정리, 볼륨 확장 |
| **백업 없음/오래됨(빨강)** | 최근 백업이 없거나 48시간보다 오래됨 | 서비스 상시 구동 여부·03:00 스케줄·디스크 확인 → **운영 점검 → [지금 백업]** 으로 즉시 생성 |
| **무결성 위반** | 자료 점검에서 실패 항목 존재 | 아래 "무결성 점검" 참조 |

## 무결성 점검

- 저장된 자료에 깨지거나 어긋난 곳이 없는지 검사합니다. 기동 직후 + 매주 자동 실행됩니다.
- 화면에서: **운영 점검 → [지금 점검 실행]** — 항목별 통과/위반이 표로 표시됩니다.
  (IT 참고: `GET /api/admin/integrity` · 즉시 실행 `POST /api/admin/integrity/run`)
- 위반 시: 화면을 캡처해 시스템 담당자에게 전달 → 위반 대상(`offenders`)을 편집 화면에서 정정.
  **자동 삭제·자동 수정은 없습니다.**

## 데모/초기화 (설치 검증용)

- 빈 `data/` 에서 데모 데이터를 시드: `npm run seed:demo`(admin/day1pass · 가상 분야 2개 · 문항 55개).
- 기존 데이터가 있으면 먼저 `data/` 를 `data.bak-<시각>/` 로 옮긴 뒤 실행하세요(안전장치가 덮어쓰기를 막습니다).

## 인증심사(실사) 전날 체크리스트

**운영 점검 화면 맨 아래에 이 목록이 체크리스트로 내장**되어 있어, 담당자는 화면에서
하나씩 체크하며 따라가면 됩니다. 전부 마우스 클릭으로 끝납니다:

- [ ] 운영 점검에서 **[지금 백업]** 을 눌러 백업 파일 생성.
- [ ] 설치 폴더의 **`data` 폴더를 USB 등 다른 장치에 복사**해 보관 (탐색기 복사→붙여넣기).
- [ ] 운영 점검에서 **[지금 점검 실행]** — 모든 항목 "통과" 확인.
- [ ] **사용자 계정 관리**에서 심사위원용 열람(viewer) 계정 발급 + 만료일 설정.
- [ ] 검색창에 문항번호를 입력해 **3초 안에 해당 문항이 열리는지** 연습.
- [ ] 분야 화면에서 **[엑셀 내보내기]** 와 인쇄가 되는지 확인.
- [ ] 서버 PC를 **재부팅한 뒤에도 접속되는지** 확인 (상시 운영 설정을 한 경우).
