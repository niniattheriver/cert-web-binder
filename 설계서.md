# 인증 문항 근거자료 관리 웹시스템 ("웹 바인더") — 구현 설계서

**버전 1.0 · 2026-07-12 · 상태: 구현 착수용 확정 설계 (자기완결 문서)**

이 문서 하나로 구현이 가능하도록 배경·요구사항·아키텍처·데이터 모델·화면·마일스톤·검증 기준을
모두 담았다. 구현 담당자는 이 문서를 유일한 사양 원천으로 삼는다.

---

## 0. 배경과 요구사항

### 0.1 배경
인증 심사 대비로 **약 10개 분야 × 분야별 ~100개 문항(총 ~1,000개)** 의 답변·근거자료를
지금까지 종이 바인더로 제작해 왔으나 시간·노동력이 과다하다. 이를 **사내 내부망 웹시스템**으로
전환한다. 심사 당일 심사위원 앞에서 프로젝터로 시연하는 상황이 핵심 사용 장면이다.

### 0.2 확정 요구사항
| 항목 | 내용 |
|---|---|
| 사용 환경 | 사내 서버/내부망. 여러 직원이 브라우저로 접속. 서버 OS 미정(Windows/Linux 모두 대비). **인터넷·CDN 접근 전무 전제** |
| 편집 주체 | 여러 담당자가 각자 분야를 편집 → 간단한 로그인 필요 |
| 지침서 PDF | 텍스트 선택 가능(스캔본 아님) → 텍스트 기반 하이라이트 가능 |
| 근거자료 유형 | ① 지침서 PDF 구간 매핑(양방향), ② 자유형식 문서(텍스트·표·이미지) |
| 문항 데이터 | 종이/PDF로만 존재 → 엑셀 템플릿 일괄등록 + 문항 PDF 자동 파싱 |
| 유지관리 | 비개발자 → 운영 단순성 최우선 (SQLite, 단일 프로세스, 폴더 복사 백업) |
| 기타 | 한국어 UI, 인쇄/내보내기(바인더 대체), PDF 내 텍스트 검색 |

### 0.3 추가 요구사항
- **(R1) 매년 문항 개정 — 문항은 전 기관 공통**: 인증 문항은 모든 기관에 동일하게 적용되는 표준.
  매년 일부 변경/삭제/추가되어 인증기관 홈페이지에 분야별 PDF로 게시(로그인 후 수동 다운로드 —
  자동 수집은 범위 밖). **각 문항 PDF 앞부분에 공식 개정 요약표(신규/수정/삭제 + 문항번호 + 수정유형)가
  포함되어 있음** — 예: "설명 수정", "문항 수정, 설명 수정, 배점 변경", "해당없음 유무변경", 빈 표는 `-`.
  문항번호 형식은 `50.010.090` 같은 점 구분 코드. 연초에 신년도 문항 PDF들을 **일괄 업로드하면
  자동 파싱 → 새 주기의 문항으로 전환**되고, 전년도 대비 **변경/신규 문항에 배지 표시**.
  기관별 자산은 지침서·답변·매핑·채점뿐이다.
- **(R2) 공모전 출품 + GitHub 공개**: 다른 기관도 자기 내부망에 설치할 수 있는 범용 오픈소스.
  기관 고유 데이터 하드코딩 금지, MIT 라이선스, 데모 데이터·설치 문서 필수.
- **(R3) 채점**: 문항별 배점(상이, 문항서에 기재). "예" 선택 시 **0.5점 간격** 채점(0~배점),
  "아니오"=0점 자동, 일부 문항은 "해당없음" 선택 가능. 문항 하단에 **지적/권장사항** 텍스트 입력칸.
- **(R4) 기간·주체**: **3일 안에 v1 완성**이 목표. 구현 담당자가 전권을 가짐 — 이 설계서는
  확정 출발점이되, 구현 중 자유롭게 수정 가능(수정 시 이 문서를 함께 갱신해 사양 원천을 유지).
  §10이 3일 계획과 백로그(v2) 구분을 담는다.

### 0.4 설계 확정 경위
3개 관점(운영 단순성 / 매핑 견고성 / UX) 독립 설계 → 심사 → 종합으로 확정.
**골격 = 견고성안**(논리 발췌/판본별 앵커 분리, 불변 판본, 재앵커링 파이프라인),
**상호작용 = UX안**(3분할 화면, 근거 칩, 무대화상자 매핑 루프, 옴니박스),
**배포 = 단순운영안**(Docker 없이 WinSW/systemd 우선, app/·data/ 분리).
심사 근거의 핵심: 이 시스템의 핵심 자산은 문항↔지침서 매핑이고, 수명 내내 반드시 반복되는 사건은
지침서 개정이다. "조용한 이관"은 심사장 프로젝터에 틀린 근거를 띄울 수 있는 유일한 경로이므로
배제하고, 불변 판본 모델은 후장착이 불가능하므로 스키마 단계부터 넣는다.

## 1. 아키텍처 / 기술 스택

**형태: 단일 Node.js 프로세스가 정적 SPA와 JSON API를 한 포트(기본 8080)로 서빙.
상태 = SQLite 파일 1개 + 파일 폴더 1개. 역프록시·Redis·큐·컨테이너 필수 없음.**

| 계층 | 선택 | 이유 |
|---|---|---|
| 런타임 | Node.js 20/22 LTS | 유일한 설치 선행조건 (Windows MSI/Linux 패키지 모두 존재) |
| 서버 | Express | 문서 많고 충분함; express-session 생태계 직결 |
| DB | SQLite(WAL) + better-sqlite3 | 파일 하나 = 백업 하나; 동기 API로 비동기 버그 차단; 프리빌트 바이너리(Windows 컴파일 불요) |
| 검색 | SQLite FTS5, **trigram 토크나이저** | 한국어는 unicode61 분절이 깨짐 — trigram 부분문자열 검색이 정답 |
| 프런트 | React 18 + TypeScript + Vite | 분할 창/뷰어 상태 관리 |
| PDF 뷰어 | pdfjs-dist + **react-pdf-highlighter-extended** (MIT) | 하이라이트 = 임의 React 컴포넌트 → 문항번호 배지·클릭 핸들러가 기본 사용법. 폴백: 텍스트레이어 위 자체 오버레이(~300줄; 데이터 포맷은 우리 소유라 교체 무통) |
| CJK 필수 설정 | **cMaps(Adobe-Korea1 계열)·standard_fonts를 앱 정적 경로에 동봉**, `cMapUrl`/`standardFontDataUrl` 로컬 지정 | 내부망 = CDN 완전 불가; cMaps 누락 시 한국어 텍스트 추출이 깨짐 → 빌드에 포함 |
| 서버측 PDF 추출 | pdfjs-dist (Node **legacy 빌드**) | 클라이언트와 동일 추출 엔진 → 좌표·텍스트 불일치 리스크 구조적 제거. AGPL(PyMuPDF) 회피 |
| 퍼지 매칭 | approx-string-match (MIT) | Hypothes.is 앵커링 스택과 동일 계열; 순수 JS |
| 리치텍스트 | **Tiptap 3** (StarterKit + Table + Image + FileHandler) | 표·이미지·워드 붙여넣기; ProseMirror JSON 저장 |
| 엑셀 | **ExcelJS** (MIT) | 가져오기·양식 생성·내보내기 전부. ⚠️ npm `xlsx@0.18.5`는 취약점 방치로 금지 |
| 인증 | express-session + SQLite 세션스토어 + bcryptjs | 내부망 단일 서버의 표준; bcryptjs는 순수 JS(네이티브 빌드 리스크 0) |
| 인쇄 | 브라우저 print CSS (1단계) | 서버측 PDF 엔진은 최대 운영 복잡성 항목 — 필요 시 2단계에 Playwright(Chromium 사전 동봉) |

**명시적 배제**: Postgres/Redis/역프록시/메시지큐(규모 대비 무의미), SSO/LDAP(요구 없음),
PDF 파일 내 주석 굽기(원본 불변성 위반), OCR(텍스트 선택형 확인됨), HWP 직접 렌더링,
@react-pdf-viewer(아카이브됨), Lucia(아카이브됨), CKEditor 5(GPL/상용), wkhtmltopdf(폐기),
PyMuPDF(AGPL). **전 의존성 MIT/Apache** — R2 오픈소스 공개에 라이선스 지뢰 없음.

## 2. 데이터 모델

SQLite, `PRAGMA foreign_keys=ON`, WAL. 핵심은 **논리 발췌(passage) / 판본별 앵커(passage_anchor)
분리** — 문항은 논리 발췌에 연결되므로 지침서 개정이 문항↔근거 관계를 절대 끊지 못하고,
현재 판본 앵커의 "해결 상태"만 바꿀 수 있다.

```sql
-- 사용자
CREATE TABLE user (
  id INTEGER PRIMARY KEY, username TEXT UNIQUE NOT NULL,
  pw_hash TEXT NOT NULL,                      -- bcryptjs
  display_name TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('admin','editor','viewer')),  -- viewer = 심사위원 열람용
  expires_at TEXT,                            -- viewer 임시계정 만료
  active INTEGER NOT NULL DEFAULT 1
);

-- 기관 설정 (R2: 하드코딩 금지 — 기관명/표시명/로고 등)
CREATE TABLE app_setting ( key TEXT PRIMARY KEY, value TEXT NOT NULL );

-- 감사 주기 = 연도별 문항 판 (R1)
CREATE TABLE cycle (
  id INTEGER PRIMARY KEY, name TEXT NOT NULL,          -- '2026년 정기심사'
  status TEXT NOT NULL DEFAULT 'active'
         CHECK (status IN ('active','frozen')),        -- frozen = 제출본 동결, 전면 읽기전용
  frozen_at TEXT, frozen_by INTEGER REFERENCES user(id),
  created_at TEXT NOT NULL,
  year INTEGER                               -- migration 007: 정식 연도 필드 —
                                             -- 연도별 홈·가져오기 대상 연도·이월 소스 판별. 이름 파싱 금지.
);
-- "현재" 주기는 app_setting 'activeCycleId'로 고정 — 미래 연도 주기를 미리
-- 만들어도 결과 요약·검색·기관 지표 등의 기본 스코프는 바뀌지 않는다. 전환 UI는 v2(주기 관리).

CREATE TABLE category (
  id INTEGER PRIMARY KEY, cycle_id INTEGER NOT NULL REFERENCES cycle(id),
  code TEXT NOT NULL, name TEXT NOT NULL,
  owner_user_id INTEGER REFERENCES user(id),           -- 소유는 표식일 뿐 ACL 아님
  sort INTEGER NOT NULL DEFAULT 0, deleted_at TEXT,
  UNIQUE (cycle_id, code)
);

CREATE TABLE question (
  id INTEGER PRIMARY KEY,
  category_id INTEGER NOT NULL REFERENCES category(id),
  question_no TEXT NOT NULL,                 -- '50.010.090' 점 구분 코드 (표시용; 정체성은 id)
  sort_key INTEGER NOT NULL DEFAULT 0,
  body TEXT NOT NULL,
  answer_json TEXT, answer_plain TEXT,       -- Tiptap JSON + FTS/내보내기용 평문 투영
  -- 채점 (R3)
  max_score REAL,                            -- 배점 (문항서 기재; 문항별 상이)
  allow_na INTEGER NOT NULL DEFAULT 0,       -- '해당없음' 선택 가능 여부
  answer_choice TEXT CHECK (answer_choice IN ('yes','no','na')),
  score REAL,                                -- 예: 0~max_score, 0.5 간격(서버 검증); 아니오: 0 자동; 해당없음: NULL
  findings_text TEXT,                        -- 지적/권장사항 (하단 입력칸)
  -- 문항 유형 (실물 서식 확정 — migration 002. 핵심 C/필요 R/기본 B. 채점 계산엔 무관, 표시·필터용)
  question_type TEXT CHECK (question_type IN ('core','required','basic')),
  grade_symbol TEXT,                         -- 원문 등급 기호(C/R/B)
  -- 통합 채점 모델 (v1.5 — migration 003. 스키마 선행 생성, 동작은 Phase 3a 장착)
  scoring_mode TEXT NOT NULL DEFAULT 'simple'
       CHECK (scoring_mode IN ('simple','composite','auto')),
                                             -- simple=예/아니오 단일 배점(현행) / composite=세부항목 합산 / auto=기관지표 자동
  topic TEXT,                                -- 문항 '주제'(설명과 분리). 파서 확장(Phase 3b) 전까지 NULL — body에서 파생 표시
  score_overridden INTEGER NOT NULL DEFAULT 0, -- 합산·자동 모드에서 총점을 수기 override 했는지 표식(사유 필수)
  score_autofilled INTEGER NOT NULL DEFAULT 0, -- 예→만점 자동 채움 후 미확인 (migration 004 — UI 액션 전용,
                                             --   임포트·마이그레이션 경로에서 절대 1로 설정 금지. '예' 외 선택 시 서버가 소거)
  -- 챕터 = 목차 대·중분류 제목 (v1.5 — migration 005, Phase 3b. 파서 소유분 — 재인입 화이트리스트)
  chapter_major TEXT,                        -- 예: "3 질관리: 일반". 목차 파싱 실패 시 NULL(UI는 번호 접두 폴백)
  chapter_minor TEXT,                        -- 예: "1. 검사지침서". 대분류 직속 문항은 NULL
  -- 연차 개정 (R1)
  revision_status TEXT CHECK (revision_status IN ('same','modified','new')),
  revision_note TEXT,                        -- 공식 개정 요약표의 수정유형 원문 ('배점 변경, 설명 수정')
  needs_recheck INTEGER NOT NULL DEFAULT 0,  -- 본문 변경/배점 변경/해당없음 유무변경 시 마법사가 설정
  auto_candidate INTEGER NOT NULL DEFAULT 0, -- 자동배점 문항 표시 (migration 006).
                                             -- 파서 소유: ① 본문 구간표(임계표) 감지 ② "자동 배점" 명시 문구.
                                             -- 실측 8문항. UI '자동배점' 배지 — auto 모드 활성화와는 별개
  carried_from_id INTEGER REFERENCES question(id),  -- 전년도 문항 행 (비교 보기의 근거)
  -- 공통
  reviewed INTEGER NOT NULL DEFAULT 0,       -- 수동 '검토완료 ✓' (완성도 ○/◐/●는 파생값, 저장 안 함)
  row_version INTEGER NOT NULL DEFAULT 1,    -- 낙관적 잠금
  updated_at TEXT NOT NULL, updated_by INTEGER REFERENCES user(id),
  deleted_at TEXT,
  UNIQUE (category_id, question_no)
);

-- ── 통합 채점 모델 + 기관 지표 (v1.5 — migration 003) ──
-- 가드레일 4(전체 스키마 선행 생성): 채점 모델은 후장착이 재작성급이라 자리를 먼저 잡는다.
-- 동작(합산 UI·자동 계산)은 v1.5 Phase 3a에서 장착.

-- 세부 평가항목 — 합산(composite) 문항의 항목별 배점/취득점. parent_id로 2단 계층 허용.
CREATE TABLE question_criterion (
  id INTEGER PRIMARY KEY,
  question_id INTEGER NOT NULL REFERENCES question(id),
  parent_id INTEGER REFERENCES question_criterion(id),   -- 중첩 항목(선택). NULL=최상위
  sort INTEGER NOT NULL DEFAULT 0,
  label TEXT NOT NULL,                                    -- 항목명(파서 추출 또는 수기)
  max_score REAL NOT NULL,                                -- 항목 배점(양수, 서버 검증)
  score REAL,                                             -- 취득점(0~max_score, 미채점 NULL)
  deleted_at TEXT
);

-- 자동배점 규칙 — 문항 1개당 0/1개. source_metric_key가 NULL이면 '미활성'(바인딩 전).
-- 파서는 감지·임계표 추출까지, 지표 바인딩·활성화는 사람이 수동으로 한다(조용한 자동화 금지).
CREATE TABLE auto_rule (
  id INTEGER PRIMARY KEY,
  question_id INTEGER NOT NULL REFERENCES question(id),
  source_metric_key TEXT,                                 -- org_metric.metric_key (수동 바인딩; NULL=미활성)
  created_at TEXT NOT NULL, updated_at TEXT NOT NULL,
  UNIQUE (question_id)
);
CREATE TABLE auto_rule_band (                             -- [lower, upper) 구간별 점수. 서버가 겹침·구멍 검증
  id INTEGER PRIMARY KEY,
  auto_rule_id INTEGER NOT NULL REFERENCES auto_rule(id),
  lower REAL, upper REAL,                                 -- NULL=−∞/+∞
  score REAL NOT NULL, sort INTEGER NOT NULL DEFAULT 0
);

-- 기관 지표 — 공통문항 입력값(전년도 검사건수·상근 전문의 수 등). cycle 스코프(주기별 상이 —
-- 전역 값이면 내년 입력이 올해 점수를 소급 파괴). 하드코딩 금지(R2): 지표는 데이터로 관리.
CREATE TABLE org_metric (
  id INTEGER PRIMARY KEY,
  cycle_id INTEGER NOT NULL REFERENCES cycle(id),
```

**무결성 불변식** (서비스 계층 + 기동 시 점검기 + 관리 대시보드 표시):
- 문항이 연결한 모든 비-obsolete passage는 해당 문서의 **현재 판본에 앵커 행이 존재**해야 함(상태 불문 — 추적이 핵심).
- 문서당 `is_current=1` 판본은 정확히 1개. 모든 판본의 sha256 파일 + **첨부 저장소 파일(문항 첨부 —
  soft delete 포함·지침서 원본·에디터 이미지)**이 디스크에 존재(주간 전수 검사 — v1.5 Phase 2 확장.
  [지금 백업] 파일 매니페스트도 동일 대상).
- **채점 정합(simple)**: `answer_choice='no'` → `score=0` / `'na'` → `score IS NULL`(allow_na=1인 문항만) /
  `'yes'` → `score IS NULL`(**미채점** — 허용되는 중간 상태) 또는 `0 ≤ score ≤ max_score`이고
  `score*2`가 정수(0.5 간격).
- **채점 정합(composite/auto — v1.5)**: `question.score`는 전 모드에서 "구체화된 유효 총점"으로 유지하되,
  합산/자동 모드에서는 **서버만 쓴다**(UI 스테퍼는 읽기 전용 합계 표시). composite → `score == Σ 리프
  criterion.score`(**리프 = 비삭제 자식이 없는 항목** — 2단 계층에서 부모 이중 합산 방지. 리프 전부
  미채점이면 NULL, 일부 채점이면 채점분 부분합 = 진행 중 표시·감점 아님),
  `max_score == Σ 리프 criterion.max_score`(파서 오추출 감지망 겸용). auto → `score == auto_score_state.computed_score`
  (수기 수정은 명시적 override — `score_overridden=1` + 사유 change_log). auto_rule_band는 겹침·구멍 없이
  검증(서버). **지표 미입력(`org_metric.value IS NULL`)은 '입력값 없음' — 0점/만점이 아니라 계산 불가**로
  결과 요약에 노출. 지표 수정 시 조용한 재계산 금지: 의존 문항 `auto_score_state.stale=1` → 검수 큐에서
  "X점→Y점" diff 원클릭 확정.
- `question_passage`/`question_richdoc`는 soft-delete된 행을 참조 불가.
- 도메인 데이터 하드삭제 없음 — soft delete + change_log. 휴지통 비우기는 관리자 명시 명령만.

## 3. PDF 매핑 동작 방식

### 3.1 업로드 → 추출 파이프라인
지침서 판본 업로드 시 서버가 동기 실행(수백 페이지 = 수 초):

1. **업로드 검증**: 암호화/열람제한 PDF → 즉시 거부("암호를 해제한 PDF로 다시 업로드하세요"),
   파싱 실패(손상) → 거부 + 원인 표시, 크기 상한 `config.json`의 `maxPdfMB`(기본 200).
   동일 sha256 재업로드는 무동작(중복 제거).
2. **추출**: pdfjs-dist(Node legacy 빌드, 로컬 cMaps 지정)로 페이지별 `getTextContent()` —
   텍스트 아이템 + transform 좌표 확보.
3. **정규화**: 유니코드 **NFC**(한글 NFC/NFD 혼입은 문자열 매칭을 깨는 1급 리스크), 제로폭 문자 제거,
   줄끝 통일. **저장 텍스트의 공백은 붕괴시키지 않음** — 오프셋 안정성 우선. 매칭은 오프셋 맵을 가진
   정규화 그림자 문자열로 수행.
4. **저장**: `page_text` + 페이지 시작 오프셋 → 전역 오프셋 ↔ (페이지, 지역 오프셋) 양방향 변환.
   아이템 geometry는 저장하지 않고(비대) 앵커 확정/재해결 시 재계산해 `rects_json`에 캐시.
5. **위생 점검**: 페이지 텍스트 밀도가 비정상적으로 낮으면(스캔 혼입 의심) `text_warning` 설정 →
   해당 페이지는 박스 앵커만 허용, 업로드 화면에 경고.
6. **FTS 인덱싱** → 앱 내 지침서 전문 검색.

### 3.2 하이라이트 생성 — 무대화상자 매핑 루프
**문항에서 출발(주경로)**: 문항 화면에서 `H` → 문서 선택(최근순, 타이핑 필터) → 뷰어가 **매핑 모드**
(얇은 주황 테두리 = "선택은 이 문항에 연결됨")로 열림 → `/`로 문서 내 검색 → **드래그 선택** →
선택부에 플로팅 툴바:

```
██████████ │ 「50.210.420」에 근거로 연결  [연결 ✓ Enter]  [색 ▾]  [취소 Esc] │
```

`Enter` → 서버가 한 트랜잭션으로 `passage` + 현재판본 `passage_anchor`(quote/prefix/suffix +
오프셋 + 클라이언트가 계산한 정규화 rects, status=`resolved`, method=`manual`) + `question_passage`
생성 → 근거 칩으로 즉시 등장, **뷰어는 그 자리 그대로 — 바로 다음 선택**. 대화상자 0개. `Esc` 종료.
3–4단어 미만 초단문 선택은 "문장 단위로 선택하면 개정 시 안전합니다" 넛지(고아화 확률 때문).

**PDF에서 출발**: 문서 뷰어에서 드래그 → 같은 툴바, [연결]이 문항 퀵피커(번호/키워드, 다중 선택) 오픈.

**중복 방지**: 새 선택이 기존 하이라이트와 **≥60% 겹치면** "기존 하이라이트에 이 문항 추가"를 기본
제안 — 한 구절에 배지가 쌓이는 모델을 깨끗하게 유지. 마지막 문항 연결이 해제된 passage는 확인 후
soft-delete(앵커·이력은 change_log에 잔존).

**표/이상 읽기순서 탈출구**: 박스 드래그 앵커(`geometry_primary=1`) — quote는 최선노력 캡처,
개정 시 **무조건 검수행**(geometry는 판본 간 자동 이관 금지).

### 3.3 양방향 표시
- **문항 → 지침서**: 문항 상세의 근거 카드(칩 확장형)에 **발췌 인용문을 인라인 미리보기로 먼저 표시**
  (지침서 해당 부분 텍스트 + 출처행 "○○지침 v2026-1, p.37") → 카드의 **[상세보기]** 클릭 시
  전체 지침서가 뷰어 창에 열리며 **해당 부분으로 즉시 스크롤 + 하이라이트 2회 펄스**(프로젝터 청중이
  즉시 인지). 문항 열람 시 근거 ①은 자동 로드. 칩 ①②③ = 숫자키 1–9.
- **지침서 → 문항 (역방향)**: 뷰어는 판본의 전 앵커를 오버레이로 렌더. 문항과 매핑된 구간은
  **하이라이트 배경색 + 굵은 밑줄** 오버레이(원본 PDF 글리프 자체를 볼드로 바꾸는 것은 PDF 렌더링
  특성상 불가 → 진한 배경+굵은 밑줄+배지로 동등 이상의 시인성 확보) + 여백 배지
  `[50.210.420 · 50.504.100 +2]`(겹침은 클러스터). **하이라이트 영역 또는 배지 클릭** → 옆에
  메모 형식 팝오버 — 이것이 요구사항의 "팝업/메모"에 해당하는 유일하게 허용된 팝업:

```
┌─ 이 근거를 인용한 문항 ──────────────────────┐
│ 50.210.420 개인정보 파기 절차               │
│   "파기 대장을 작성하고…" (내용 미리보기)  [열기 ↗] │
│ 50.504.100 위탁 종료 시 자료 처리            │
│   "위탁계약 종료 후…" (내용 미리보기)      [열기 ↗] │
│ [+ 다른 문항에도 연결]  [연결 해제…]           │
└──────────────────────────────────────────────┘
```

- 우측 레일 = 문서 전체 매핑 색인(페이지순, 인용 스니펫, 문항번호) — "37페이지에 뭐가 걸려 있죠?"에
  즉답. 딥링크 `/docs/:id?hl=…`. 판본 드롭다운으로 **과거 판본을 당시 앵커 그대로** 열람.

### 3.4 지침서 개정 — 재앵커링
새 판본 업로드가 자동 트리거. 직전 현재판본의 `resolved*` 앵커 각각에 대해:

```
expected = old.start / old_doc_len × new_doc_len          # 위치 사전확률

1단계 — 정확 일치: 정규화 그림자에서 quote_exact 전건 탐색
   1건        → resolved_auto (confidence 1.0)
   여러 건     → prefix/suffix 유사도 + 위치 근접도로 점수화;
                차점과 여유 있으면 resolved_auto(≈0.95), 아니면 needs_review(후보목록 첨부)
2단계 — 퍼지(approx-string-match): expected 주변 반경 max(4000자, 문서의 10%) 창, 미발견 시 전문 확장
   유사도 ≥0.90 → resolved_fuzzy (원클릭 확인 필요)
   ≥0.60       → needs_review (후보 프리필)
   미만         → unresolved (조항 삭제/전면 개정 추정)
3단계 — 해결된 앵커의 오프셋 → page_map으로 페이지 환산, 텍스트 아이템 보간으로 rects 재계산·캐시
```

**절대 원칙: 무엇도 버리지 않고, 무엇도 조용히 넘어가지 않는다.** `resolved_fuzzy`조차 확인 전까지
대시보드 빨간 카드에 잡힌다. 중복 상용구가 흔한 지침서에서 조용한 오앵커는 심사장에 틀린 근거를
띄우는 유일한 경로다.

- 판본별 **이관 보고서** 영구 저장: "지침서 v2026-1: 214건 — 자동 187 · 확인필요(퍼지) 19 · 모호 5 · 미해결 3".
- 구판 앵커는 `historical` 보존 — 이관 중에도 "v2024 기준 근거"를 항상 제시 가능.
- 앵커 행에 `method/confidence/resolved_by/resolved_at` 기록, 전이는 전부 change_log.
- **검수 큐 화면**: 좌 = 구판 페이지+구 하이라이트, 우 = 신판 후보 위치(펄스, 모호 건은 점수순 후보
  목록). 액션 **확인(Enter) · 다시 선택(신판에서 드래그) · 근거 무효화(조항 삭제 — passage.obsolete=1,
  연결 문항 전부에 재검토 플래그) · 건너뛰기**. `J/K` 이동 — 20–30건 확인은 몇 분 작업.
  각 건에 연결 문항번호 표시(영향 반경 인지 후 판단).
- **추출기 업그레이드**: pdfjs-dist 메이저 갱신 시 `extractor` 라벨 변경 → 관리 명령이 동일 파일을
  재추출해 전 앵커를 같은 알고리즘으로 재검증(구 텍스트→신 텍스트), 이동분 보고.

원클릭 확정.
하단 지적/권장사항은 **내용 존재 여부에서 표시 파생**(내용 있으면 펼침, 없으면 [+추가] 버튼 —
별도 체크박스·불리언 저장 금지). 자동 저장·change_log 기록.
첨부파일·링크(§2 question_attachment·question_link)는 별도 하단 섹션이 아니라 **근거 자료 카드 안에
통합 표시**된다(카드 헤더 버튼 [+ 지침서 연결][+ 파일 첨부][+ 링크 추가][+ 자유형식],
칩 순번 ①②③에는 편입하지 않음: 칩은 '본문 근거', 첨부·링크는 '보조 자료'라는 §2 원칙 유지).
자유형식 문서 근거 칩은 우측 페인에 **읽기전용 대형 표시**(편집은 /rich/:id), PDF 하이라이트 여백
배지에는 현재 문항의 **근거 칩 순번 ①②③이 병기**된다.

**단축키 전도**(`?` 오버레이): `Ctrl+K·/` 검색 · `j/k` 문항 이동 · `1–9` 근거 칩 ·

## 5. 권한 · 로그인 · 동시편집

- **세션**: express-session + better-sqlite3 스토어, httpOnly SameSite=Lax 쿠키, 12시간 유휴 만료.
  bcryptjs 해시. 자가가입 없음 — 관리자 생성. 내부망 HTTP 기본이므로 `secure` 플래그는 config 조건부.
  HTTPS 리버스 프록시 뒤에서는 config `trustProxy`(Express 'trust proxy' 값)를 `secureCookies`와
  함께 설정해야 secure 쿠키가 정상 발급된다(무조건 신뢰 금지 — 홉 수/주소로 제한. `true` 는
  X-Forwarded-* 위조로 로그인 잠금·secure 판정이 무력화되므로 기동 시 거부).
- **로그인 방어(v1.5.1)**: 사용자명+IP 조합 연속 실패 5회 → 60초 잠금(429 + Retry-After).
  비밀번호 변경은 본인 외 세션 전부 무효화, 관리자 재설정은 대상 세션 전부 무효화(세션 테이블
  `json_extract(sess,'$.userId')` 삭제). 공통 보안 헤더(nosniff·X-Frame-Options DENY·Referrer-Policy).
- **역할 3종**: `admin`(사용자·주기 동결·판본 교체·정리·삭제) / `editor`(그 외 전부) /
  `viewer`(열람 전용). **분야 소유는 표식** — 타인 분야 편집 시 확인 토스트("박담당 분야입니다 —
  계속하시겠어요?")만, ACL 없음. 책임 추적은 change_log가 담당.
- **동시편집 2중 방어**:
  1. **낙관적 버전(권위층)**: 모든 저장이 로드 시점 `row_version` 동봉, 불일치 → 409 + 서버본.
     UI는 내 텍스트/서버 텍스트 좌우 비교 + [최신 불러오기(권장)] / [내 내용으로 덮어쓰기(명시적,
     로그 기록)] / [내 내용 복사해두기]. **지는 쪽 텍스트도 change_log에 보존** — 조용한 유실 불가.
  2. **권고 리스(예의층)**: 편집 진입 시 리스(TTL 2분, 30초 하트비트). 타인에게 목록 행 칩(✎김대리)과
     편집화면 배너로 표시. 브라우저 사망이 누구도 잠그지 못함.
- **가져오기-편집 동시성**: (a) 가져오기 커밋 직전 대상 문항에 활성 리스가 있으면 목록 표시 후 확인
  요구. (b) 커밋은 대상 행 `row_version`을 정상 증가시키고 change_log에 `actor_kind='import',
  batch_id` 기록 → 편집자의 다음 저장은 일반 409로 잡히되, 충돌 메시지가 "엑셀 가져오기(배치 #12,
  실행자 김담당, 10:32)로 변경되었습니다"로 귀속을 명확히 표시.

> 지정**으로 장착 — 드라이런 multipart에 `year`(2000..2100)·`carry`('1' 기본) 필드, 대상 주기는
> `cycle.year`로 찾고 없으면 커밋 시 생성(`{year}년 심사`, **activeCycleId 불변** — 현재 연도
> 유지). 이월 소스 = 대상보다 작은 최대 연도의 주기. 드라이런 리포트에 carryMatched/
> carryWithAnswer/carryWithEvidence, 커밋 결과에 carried* 집계 + change_log `action='carry'`.
> 주기 동결·전환 UI는 v2 유지.

- **변경/신규 표시**: 문항 목록·상세에 `신규`/`변경됨` 배지(**툴팁에 공식 수정유형** — "배점 변경,
  설명 수정"), "변경·신규만 보기" 필터(연초 할일 목록), 변경 문항 상세에서 [전년도와 비교] 나란히 보기.
- **폴백**: 서식이 파서 예상과 다르면 파싱 결과를 엑셀로 내려받아 수정 후 재업로드, 또는 처음부터
  §6.1 엑셀 경로 사용(동일 diff 파이프라인 공유).
- **문항 팩 공유(R1×R2)**: 확정된 주기의 문항 세트를 답변 없이 파일(JSON/엑셀)로 내보내기/가져오기 —
  한 기관이 정리한 신년도 문항을 타 기관이 가져와 자기 근거만 붙이면 됨. 실제 인증 문항은 인증기관
  저작물이므로 **공개 저장소에는 미포함**(가상 데모 문항만 동봉).
- 인증기관 배포 문항 원본 PDF는 `document(kind='question_source')`로 등록해 앱 안에서 참고 열람 가능.

### 6.3 엑셀 왕복 내보내기
`/admin` 및 분야 화면에서 ExcelJS로:
`분야코드 | 문항번호 | 문항 | 배점 | 선택(예/아니오/해당없음) | 점수 | 지적/권장사항 | 답변(평문) |
상태 | 근거요약(발췌 출처행·문서 제목) | 최종수정자/일시` — **분야별 합계 행**(취득/만점/달성률) 포함.
경영보고·부서 회람·제출 서류용. 점수 열은 채점 방식(§2 scoring_mode) 무관하게 구체화된 유효 총점.
**전체 내보내기에는 '준비도 요약' 시트 1장 추가**(v1.5 Phase 3a — C-2, 심사 D-7 점검 회의용):
분야별 근거 없음(발췌·자유형식·첨부파일·링크 어느 것도 없는 문항 — 첨부·링크가 근거 자료 카드에
통합된 이후 '근거 있음'으로 인정. 단 칩 순번 ①②③은 여전히 본문 근거만)·자동입력 미확정·
재확인 필요·지표 미입력(자동배점인데 지표 미바인딩 또는 값 미입력) + 전체 행(확인 필요 미처리 병기).

### 6.4 인쇄 / 오프라인 꾸러미
- **인쇄 뷰 `/print/:categoryId`** (범위 지정 `?q=…` 가능): 표지·목차 → 문항별로 번호·본문·배점·
  선택·점수 → 답변 → 근거: **하이라이트 영역 크롭 이미지**(PDF.js 캔버스 렌더를 클라이언트에서 크롭,
  하이라이트 표시+여백 포함 — 바인더급 시각 충실도) + 인용문 + 출처행 → 지적/권장사항 → 자유문서
  전문(표·이미지 포함). `@media print`, `break-inside: avoid`, `@page` 여백, 바닥글 "분야 3 — n/N".
  **내보내기 = 브라우저 인쇄→PDF, 서버 의존성 0.** 파일형 공식 PDF가 필요해지면 2단계로
  Playwright(Chromium을 배포 zip에 사전 동봉, `PLAYWRIGHT_BROWSERS_PATH` 고정).
- **오프라인 감사 꾸러미**: `/admin` → [감사 꾸러미 내보내기] — 자기완결 폴더 zip: 분야별 정적
  HTML(인쇄 뷰와 동일 렌더) + `pdfs/` 원본 + 로컬 pdfjs 뷰어 번들(CDN 0). 노트북에 풀어 두면
  **서버·회의실 네트워크가 죽어도 심사 진행**. 운영 문서에 "심사 D-1 꾸러미 내보내기 리허설" 체크리스트.

## 7. 배포 · 백업 (Windows/Linux 내부망)

**Docker는 기본 경로가 아니다.** 서버 OS 미상 + 비개발자 유지보수 = Docker-on-Windows가 최대 운영 리스크.

```
C:\qaevidence\            또는  /opt/qaevidence/
├── app/                  # 릴리스 zip: 서버 번들 + 빌드된 SPA(+ cMaps/standard_fonts 동봉)
│                         #             + node_modules. 업그레이드 = 이 폴더만 교체
├── data/                 # 모든 상태. 업그레이드가 절대 건드리지 않음
│   ├── app.db (+ -wal/-shm)
│   ├── files/sha256/ab/cdef….pdf     # PDF·이미지, 내용주소·불변
│   └── backups/          # 야간 스냅샷, pre-import 스냅샷
└── config.json           # { "port":8080, "dataDir":"../data", "maxPdfMB":200 }
                          # 세션 시크릿은 최초 기동 시 data/에 자동 생성
```

- **선행조건 단 하나: Node LTS 설치**.
- **Windows**: 릴리스 zip에 WinSW 동봉 → `qaevidence-service.exe install` → Windows 서비스(자동시작·자동재시작).
- **Linux**: 동봉된 `qaevidence.service` → `systemctl enable --now qaevidence`.
- (Docker 선호 조직용 Dockerfile 동봉하되 문서상 2순위. `npm ci --include=dev` — 웹 빌드·tsx 런타임이
  devDependency이므로 NODE_ENV=production 이어도 dev 포함 설치.)
- **업로드 가드(v1.5.1)**: 문항 인입은 파일당 32MB·최대 16파일(multer — Content-Length 없는 청크
  전송 포함 최악 총 512MB 상한), 지침서는 파일당 maxPdfMB·1파일. 여기에 Content-Length 선검사
  (문항 512MB, 지침서 maxPdfMB+32MB)가 1차 방어선으로 과대 요청을 메모리 적재 전에 413 거절.
- **버전 정책(v1.5.1)**: git 태그 = 루트/워크스페이스 package.json 버전 = 배포 zip 파일명 = /api/health
  버전. 발행된 태그는 이동하지 않고 수정 릴리스는 새 태그(vX.Y.Z+1). `engines.node >=20.19 <21 || >=22.12`.
- **CI(v1.5.1)**: `ci.yml` — main push/PR마다 typecheck + 서버 테스트 + 웹 빌드. Dependabot(주간).
- **업그레이드**: 서비스 중지 → `app/` 교체 → 시작. 마이그레이션(번호 붙은 .sql,
  `PRAGMA user_version` 추적)은 기동 시 자동 적용·기록.
- **백업**: **앱 자체가 매일 03:00** SQLite 온라인 백업(`VACUUM INTO`)으로
  `data/backups/app-YYYYMMDD.db` 생성(일 30 + 월 12 보존) — cron/작업 스케줄러 설정 불요.
  관리 화면 **[지금 백업]** = DB 스냅샷+파일 매니페스트 ZIP. 회사 측 절차는 한 줄:
  **`data/` 폴더를 다른 장비로 복사**(robocopy/rsync — 내용주소·불변이라 자연 증분). 복원 = 폴더 되돌리고 시작.
- **저장공간**: 대시보드 디스크 게이지(경고 임계 config). `/admin` **정리 마법사**: superseded 판본 중
  historical 앵커 참조 없는 파일·고아 attachment를 크기와 함께 나열 → 관리자 명시 선택·확인 →
  로그 남기고 삭제. **자동 삭제는 없다.**
- **운영 문서(한국어 1장)**: 서비스 재시작법 / data/ 위치와 복사법 / 사용자 추가 / 대시보드 빨간 카드의
  의미 / 심사 D-1 체크리스트(백업 ZIP + 오프라인 꾸러미 리허설).

## 10. 구현 계획 — 3일 v1 + 백로그(v2)

원칙 2가지는 기간과 무관하게 유지한다:
1. **최대 리스크(실물 PDF 충실도·문항 파서)는 Day 1 오전에 실증** — 실패 시 폴백을 그 자리에서 확정.
2. **전체 스키마(§2)는 Day 1에 전부 생성** — 테이블을 만드는 비용은 싸고, 불변 판본·앵커·change_log를
   나중에 후장착하는 비용은 재작성급이다. **v1에서 동작을 다 만들지 않아도 스키마는 다 만든다.**
   백로그 항목들이 스키마 변경 없이 얹히도록.

### Day 1 — 기반 + 데이터 인입
- **오전 스파이크**: ① 실물 지침서 PDF 2~3부를 react-pdf-highlighter-extended(+로컬 cMaps)에 로드 —
  한국어 텍스트 선택·검색·하이라이트 확인 (실패 시 자체 오버레이 ~300줄 폴백 확정).
  ② 올해분 문항 PDF 2~3부 파싱 — 문항번호(`50.010.090`)·본문·배점·해당없음 + 개정 요약표 추출
  (실패 시 엑셀 템플릿 경로 확정).
- 프로젝트 뼈대(§9) + **전체 스키마 마이그레이션** + 로그인/세션 + admin 시드.
- **문항 인입**: 스파이크에서 확보한 파서로 **올해분 문항 PDF 전체를 일괄 등록**(배점·해당없음 자동 —
  1,000문항 수기 입력 회피가 파서의 최대 가치). 파서가 부실한 분야만 엑셀 템플릿(§6.1) 폴백.

### Day 2 — 핵심 가치: PDF 매핑 + 문항 화면

- 뷰어 + 매핑 루프(§3.2: 드래그→플로팅 툴바→Enter 연속 매핑) + 오버레이(배경+굵은 밑줄) +
  배지·팝오버(§3.3) + 양방향 점프 + 겹침 60% 제안.

- **Day 2 종료 기준: 문항↔PDF 양방향 + 채점이 시연된다 (심사일 3초 룩업 재현).**

## 11. 검증 기준

**v1 (3일 내):**
1. **Day 1 오전 스파이크가 최우선 검증** — 실물 지침서·문항 PDF로 한국어 텍스트레이어·하이라이트·파서 실증.
2. Day 1: 실제 문항 PDF 일괄 파싱으로 등록된 문항 수·배점·해당없음 여부를 원본 PDF와 표본 대조(분야당 5~10문항).
3. Day 2: "Ctrl+K→번호→Enter→근거 펄스" 3초 룩업 재현 + PDF 하이라이트 클릭→문항 팝오버→왕복.
4. 채점: 0.5 간격·배점 상한·아니오=0 자동·해당없음(allow_na만)·집계 분모 제외를 서버 검증+화면으로 확인.
5. 동시편집: 두 브라우저 세션으로 동일 문항 저장 → 409 비교 UI·로그 보존 확인.
6. Day 3: 대상 서버/PC에 실설치 후 다른 PC 브라우저에서 접속·전 시나리오 재확인.

**v2 (백로그 구현 시):**
7. 재앵커링: 지침서 v1 매핑 → 문구 수정한 v2 업로드 → resolved_auto/needs_review(→퍼지 도입 시 fuzzy 포함) 상태 전이 확인.
8. 연차 개정: 문항 PDF 일괄 업로드 → 개정 요약표 → 변경/신규 배지·revision_note·이월·needs_recheck 종단 확인.
9. 범용성: 클린 머신에서 클론 → `seed:demo` → 데모 데이터만으로 전 기능 동작 + 저장소·git history 내 실데이터 0건.
